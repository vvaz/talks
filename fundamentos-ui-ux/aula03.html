Um processo de design UX tem em consideração:

  • User flow: Definindo que caminhos o utilizador percorre para completar as ações
  • Wireframes: São os primeiros esboços que definem que áreas do site existem e para onde são distribuidos os conteúdos
  • Low fidelity Prototype: Baseado nas conclusões dos Wireframes, este passo representa o primeiro estudo de cores e elementos visuais a serem utilizados no layout.
  • High Fidelity Mockup (ou o design final): É a versão final do layout. Todos os elementos visuais estão definidos e mostrará funcionalidades do site, embora só em imagem.

Os padrões de Web design responsivo estão a evoluir rapidamente, mas há muitos padrões estabelecidos que funcionam bem em dispositivos móveis e computadores.

Ao longo do nosso dia utilizamos diversos modos de navegação, organização e categorização da informação que consultamos em websites e aplicações. Segundo Richard Saul Wurman, podemos usar o método de organização LATCH

  • Location
  • Alphabet
  • Time
  • Category
  • Hierarchy
A informação pode ser infinita, no entanto, a organização da informação é finita, e pode ser apenas organizada pelo "LATCH"
"“Information may be infinite, however…The organization of information is finite as it can only be organized by LATCH: Location, Alphabet, Time, Category, or Hierarchy.”"

Location:

Organiza informação vindas de atlas, guias turisticos, partes de sistemas (por exemplo, localização dos pulmões, no corpo humano como mapa)

Alphabet

Utilizado quando temos muito texto para apresentar - nem sempre é o melhor método de organização de conteúdo

Time

Organiza conteúdo por eventos que ocorrem cronologicamente. Uma história, uma anedota, uma narrativa

Category

Agrupa conteúdo de importância similar, como se vê muito no eCommerce

Hierarchy

Atribui um valor ou peso à informação, por exemplo, ordenação por preço, ordenação do maior para o menor, etc.

Perguntas a fazer para organizar o conteúdo:

  • Quão grande é o conteúdo?
  • Quanto espaço é necessário para o mostrar?
  • Tem ordem?
  • O Utilizador pode pesquisar ou ordenar?
  • Que operações o utilizador pode realizar?

Mobile Direct Access
  • O ecrã principal deve accionar informação sem requerer um input por parte do utilizador
  • Deve ser usado quando a nossa App tem uma função macro genérica
  • Produzir valor instantâneo ao utilizador, criar um engagement imediato
Streams and Feeds
  • Um contínuo update de histórias, imagens, artigos, comentários, vertical scroll
  • Deve ser usado quando o nosso produto é atualizado frequentemente (ao dia)
  • Assegura que o utilizador vê sempre os elementos mais recentes
Media Browser
  • Uma grelha de elementos na qual é possivel procurar e seleccionar
  • Deve ser usado quando necessitamos de duas vistas principais
Dashboard
  • É sempre o ecrã que o utilizador vê assim que está na sua área de trabalho
  • Deve ser usado quando precisamos de fornecer um rápido feedback do ponto de situação das informações do produto
  • Página deve assegurar que se faz acompanhar de alguns elementos auxiliares como titulos, tabs, paineis, listas, gráficos, etc...
Wizard
  • Componente que direcciona o utilizador através da interface, passo a passo até concluir um processo
  • Deve ser usado quando o processo tem muitos passos.
Settings Editor
  • Página onde se encontram todas as definições, preferências ou propriedades de um sistema
  • Deve ser usado quando estamos a trabalhar num produto com muitas funcionalidades
  • Deve estar organizados com uma sequência lógica.
Modal panel
  • Um painel que se sobrepões ao ecrã corrente sem qualquer tipo de navegação
  • Normalmente é invocado pelo utilizador e vai puxar toda a sua atenção
Progress Indicator
  • Identificação clara da fase em que o Utilizador se encontra
  • Usado juntamente com o Wizard para marcar posição
Tabs
  • Conteúdo disperso por diferentes tabs
  • Deve ser utilizado quando queremos apenas disponibilizar uma informação específica de cada vez.
  • Estão amplamente utilizadas, desktops e mobile.
Accordion
  • Módulos que se expandem de acordo com a interação do utilizador
  • Deve ser utilizado quando o conteúdo é semelhante
Collapsible panels
  • Quando existe informação adicional para explicar o conteúdo principal.
  • Simplifica a interface e o utilizador consegue controlar a sua visualização.

Radio buttons

  • Consome mais espaço

Dropdown

Toggle buttons

  • Trabalha bem o espaço disponível e todas as opções estão visíveis

Single Selection list

  • Trabalha bem o espaço disponível
  • Só é visível uma escolha de cada vez.

Spinner

  • Necessita de pouco espaço
  • Requer alguma habilidade de interação

Dropdown list

  • Ocupa pouco espaço
  • Só é visivel uma escolha de cada vez, e se tiver muitos items, obriga a fazer scroll dentro da dropdown

Cascading List

Checkboxes

Multiple Selection Dropdown

  • Nem todas as opções estão visíveis.

List builder

Design Paterns Input pelo utilizador

Forgiving Format

  • Permite que o utilizador insira o formato que pretende
  • Tem o objetico de reduzir a fricção com o utilizador

Structured Format

  • Usado para obter especificamente um dado com uma formatação específica

Fill in the blanks

  • Ajuda que a interface seja muito fácil de entender

Input hints

  • Para além do placeholder text, existe uma "dica" presente
  • Esclarece o utilizador para o que tem de preencher

Responsive Enabling

  • Apenas se mostra o campo a seguir quando o utilizador preencher o campo actual.
  • Guia o utilizador num processo de inserção de dados
